# 源码分析军团

## 目标

从一个高 star 开源项目中，提炼出：
- **核心理念**：作者在设计时秉持的价值观
- **设计亮点**：那些让你说"原来还可以这样"的决策
- **可学习模式**：可以直接用到自己项目里的思路
- **权衡取舍**：每个选择背后的收益与代价

---

## 分析流程

### 步骤1：快速扫描，建立认知

**执行扫描：**
```bash
ls -la
cat README.md 2>/dev/null | head -100
cat package.json 2>/dev/null || cat go.mod 2>/dev/null || cat Cargo.toml 2>/dev/null
find . -maxdepth 2 -type d | head -30
```

**建立初步认知：**
1. 这个项目是什么？解决什么问题？
2. 作者为什么做这个项目？（从 README/Issue/提交历史找线索）
3. 它的差异化定位是什么？

### 步骤3：深度分析，提炼价值

**分析方法：**

1. **找理念的证据链**
   - 从 README/文档 找作者的明确主张
   - 从代码结构 找理念的隐性体现
   - 从 commit history 找设计演进的思路

2. **找设计的权衡点**
   - 每个设计选择都意味着取舍
   - 问：为什么选 A 不选 B？得到了什么？放弃了什么？

3. **找可学习的模式**
   - 这个模式解决了什么问题？
   - 我在什么场景下可以用？
   - 有什么注意事项？

### 步骤4：输出报告

每篇报告必须包含 Obsidian YAML frontmatter（位于文件最顶部），格式如下：

```markdown
---
title: [项目名] 源码洞察
aliases:
  - [项目名]
  - [项目名缩写/别名（如有）]
tags:
  - 开源分析
  - [主技术栈，如 Python / Rust / TypeScript]
  - [项目类型，如 AI-Agent / CLI工具 / Web框架 / 基础设施]
  - [核心领域标签，如 LLM / 工具调用 / 消息平台]
github: https://github.com/[owner]/[repo]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

**标签选取原则：**
- `开源分析` — 固定标签，所有分析文档必须有
- 主技术栈 — 项目主要编程语言
- 项目类型 — 参考：`AI-Agent`、`CLI工具`、`Web框架`、`基础设施`、`开发工具`、`消息中间件`
- 核心领域 — 2-4 个最能描述项目定位的标签
- GitHub 地址 — 必填，从 README 或 git remote 中获取

## 分析心法

**好的分析要回答三个问题：**
1. **所以呢？** — 每个技术细节都要说明对读者的价值
2. **为什么？** — 不只说做了什么，要说为什么这样做
3. **然后呢？** — 读者看完能做什么？能用在什么地方？

**分析时要警惕：**
- 不要罗列读者自己能看的信息（目录结构、文件数量、技术栈列表）
- 不要只描述"代码做了什么"，要讲"设计背后的思考"
- 不要面面俱到，要聚焦真正有价值的亮点

**高 star 项目值得深挖的地方：**
- 作者反复强调的理念 → 通常是最核心的设计哲学
- 与众不同的做法 → 通常是最有价值的创新点
- 复杂模块的简化设计 → 通常有巧妙的抽象思路
