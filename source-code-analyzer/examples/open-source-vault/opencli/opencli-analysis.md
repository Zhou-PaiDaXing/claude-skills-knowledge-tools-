# OpenCLI 源码洞察

## 一句话本质
> OpenCLI 是一个**网站 CLI 化工具**，它的核心创新是：**用声明式 YAML 把任意网站封装成命令行命令，运行时零 AI 成本，复用用户真实登录态**。

---

## 核心理念

### 作者在设计时秉持的价值观

**理念1：运行时零 AI 成本——AI 只在"生成"阶段介入**

- **体现在哪里**：README 明确写道 "Zero LLM cost — No tokens consumed at runtime"，AI 只在 `explore`/`synthesize` 阶段生成适配器
- **为什么重要**：每次调用 AI 都有成本和延迟，确定性程序可以无限次运行而零成本
- **如何借鉴**：把 AI 放在"设计/生成"阶段，产出静态配置；运行时只执行确定性逻辑

**理念2：复用用户真实登录态，而非管理凭据**

- **体现在哪里**：`strategy: cookie` 模式在用户 Chrome 内执行 `fetch({ credentials: 'include' })`，凭据永不离开浏览器
- **为什么重要**：管理用户密码/Token 是巨大的安全负担，复用现有登录态既安全又省事
- **如何借鉴**：当你需要访问需要认证的网站，问自己"能不能借用用户已有的登录态？"而非"怎么让用户给我凭据？"

**理念3：声明式优先，命令式兜底**

- **体现在哪里**：71 个适配器中估计 80%+ 是纯 YAML，只有复杂交互才用 TypeScript 实现
- **为什么重要**：声明式 YAML 易读、易生成、易维护；命令式代码强大但复杂
- **如何借鉴**：设计 DSL 或配置格式覆盖 80% 场景，剩余 20% 才用代码扩展

**理念4：最小权限编码——把安全策略声明在接口里**

- **体现在哪里**：每个适配器有 `strategy: public | cookie | session` 字段，系统据此决定是否启动浏览器
- **为什么重要**：权限边界在 YAML 层面即可审计，不需要浏览器的命令不会启动浏览器
- **如何借鉴**：把权限/安全策略编码进配置声明，而非运行时判断

**理念5：AI Agent 优先的产品定位**

- **体现在哪里**：README 写道 "Built for AI Agents"，`CLI-EXPLORER.md` 专门为 AI 生成命令列表
- **为什么重要**：AI Agent 需要确定性、结构化的工具接口，而非模糊的自然语言
- **如何借鉴**：如果你的工具可能被 AI 调用，提供机器可读的命令清单和结构化输出

---

### 与主流方案的哲学差异

| 维度 | 主流做法 | OpenCLI 的做法 | 背后的思考 |
|------|---------|---------------|-----------|
| AI 角色 | 运行时调用 LLM | 生成时用 AI，运行时零成本 | 确定性 > 灵活性，成本可控 |
| 认证方式 | 存储用户凭据 | 复用 Chrome 登录态 | 安全 + 简单 |
| 适配器编写 | 写代码 | 写 YAML（80%场景） | 降低门槛，AI 可生成 |
| 权限策略 | 运行时判断 | 声明在 YAML 里 | 可审计，最小权限 |
| 目标用户 | 人类开发者 | AI Agent + 人类 | 结构化输出优先 |
| 浏览器使用 | 默认启动 | 按需启动 | 不需要的就不开 |

## 设计亮点

### 亮点1：Strategy 字段的最小权限路由

**问题背景：**
- 有些网站完全公开，不需要登录
- 有些网站需要登录才能访问数据
- 有些网站需要完整 UI 自动化
- 传统做法：所有命令都启动浏览器，"以防万一"

**创新做法：**

用一个 `strategy` 字段声明访问策略：

```yaml
# 完全公开，纯 HTTP fetch
strategy: public
browser: false

# 借用 Cookie，在浏览器内执行 fetch
strategy: cookie
browser: true

# 完整 UI 自动化
strategy: session
browser: true
```

系统根据 `strategy` + `pipeline` 步骤自动判断是否需要启动浏览器：

```
strategy: public + 无浏览器步骤 → 纯 HTTP（零浏览器开销）
strategy: cookie + 浏览器步骤 → 启动浏览器（借用登录态）
strategy: session → 完整浏览器自动化
```

**为什么有效：**
- 最小权限原则：不需要浏览器的命令不会启动浏览器
- 成本优化：纯 HTTP 请求比浏览器快 10x+
- 安全审计：权限边界在 YAML 层面可见

**代价与适用性：**
- 代价：需要维护 `BROWSER_ONLY_STEPS` 集合
- 适用：任何"不同操作需要不同权限级别"的系统

---

### 亮点2：YAML Pipeline 声明式适配器

**问题背景：**
- 为每个网站写爬虫代码成本高
- AI 生成的代码质量不稳定
- 维护大量爬虫脚本是噩梦

**创新做法：**

用 YAML 声明数据处理流程：

```yaml
pipeline:
  - fetch:
      url: https://api.example.com/items
  - select: data.items
  - filter: item.active
  - map:
      id: ${{ item.id }}
      title: ${{ item.title }}
  - limit: 20
```

每一步是一个独立操作，数据从上往下流动。模板语法 `${{ expr }}` 支持变量插值。

**为什么有效：**
- 声明式：描述"做什么"，不描述"怎么做"
- 可组合：步骤可自由组合，复用性强
- AI 友好：AI 生成 YAML 比生成代码更稳定
- 可读：非程序员也能看懂和修改

**代价与适用性：**
- 代价：表达力有限，复杂逻辑需要 `evaluate` 内联 JS
- 适用：任何"数据提取/转换"场景，特别是 AI 生成配置

---

### 亮点3：运行时 vs 生成时的职责分离

**问题背景：**
- 很多 AI 工具在运行时调用 LLM
- 每次运行都有成本和延迟
- 输出不确定性高

**创新做法：**

明确分离两个阶段：

**生成时（一次性，有 AI 成本）：**
```
explore <url>     → 发现 API 端点
synthesize        → 生成 YAML 适配器
cascade           → 自动探测最优策略
```

**运行时（无限次，零 AI 成本）：**
```
opencli hackernews top → 执行 YAML Pipeline
opencli bilibili feed  → 执行 YAML Pipeline
```

生成的 YAML 是静态配置，之后所有运行都是确定性执行。

**为什么有效：**
- 成本可控：AI 成本只在生成时发生一次
- 确定性：同样输入，同样输出，可测试
- 速度快：运行时没有 LLM 调用延迟

**代价与适用性：**
- 代价：网站改版需要重新生成适配器
- 适用：任何"配置生成后重复使用"的场景

---

### 亮点4：Step Registry 插件模式

**问题背景：**
- 内置步骤无法覆盖所有需求
- 用户需要扩展自定义操作
- 传统做法：fork 项目，加代码，提 PR

**创新做法：**

提供一个简单的注册接口：

```typescript
// 插件中注册自定义步骤
registerStep('my-step', async (page, params, data, args) => {
  // 自定义处理逻辑
  return transformedData;
});
```

YAML 中直接使用：
```yaml
pipeline:
  - my-step: { param: value }
```

**为什么有效：**
- 开放扩展：用户无需 fork 核心代码
- 隔离性好：插件代码独立，不影响核心
- 标准化：所有步骤遵循同一接口

**代价与适用性：**
- 代价：接口设计需要前期投入
- 适用：任何需要"用户扩展功能"的平台

---

### 亮点5：Micro Daemon + Chrome Extension 架构

**问题背景：**
- CLI 是短生命周期的进程
- 浏览器是长生命周期的应用
- 如何让 CLI 控制用户的 Chrome？

**创新做法：**

```
CLI（短生命周期）
    │ HTTP POST /command
    ▼
Micro Daemon（后台常驻）
    │ WebSocket
    ▼
Chrome Extension（浏览器内）
    │ CDP
    ▼
Chrome 页面（用户登录态）
```

**关键设计：**
- Daemon 自动启动，空闲 5 分钟自动退出
- Extension 一键安装，自动连接 Daemon
- CLI 与 Daemon 通过 HTTP 通信，与 Extension 解耦

**为什么有效：**
- CLI 无需启动浏览器，响应快
- 复用用户已有的 Chrome 实例和登录态
- Daemon 自动管理，用户无感知

**代价与适用性：**
- 代价：需要用户安装 Chrome 扩展
- 适用：任何"CLI 需要控制桌面应用"的场景

## 可学习的模式

### 模式1：Strategy 字段的最小权限编码

**解决的问题：** 如何让权限边界在配置层面可见、可审计？

**核心做法：**

```yaml
# 适配器声明需要的权限级别
strategy: public | cookie | session

# 系统据此决定是否启动浏览器
if strategy == 'public' and no_browser_steps:
    use_http_fetch()
else:
    start_browser()
```

**适用场景：**
- API 调用（不同认证级别）
- 文件访问（读/写/执行）
- 数据库操作（查询/修改/管理）

---

### 模式2：运行时零 AI 成本模式

**解决的问题：** 如何利用 AI 的能力，同时避免运行时成本？

**核心做法：**

```
[生成阶段] AI 探索网站 → 生成静态配置
         ↓
[配置存储] YAML 文件保存到磁盘
         ↓
[运行阶段] 纯确定性执行，无 AI 调用
```

**适用场景：**
- 配置生成（AI 生成，程序执行）
- 代码生成（AI 写代码，编译器编译）
- 流程自动化（AI 设计流程，引擎执行）

---

### 模式3：YAML Pipeline 声明式数据处理

**解决的问题：** 如何让非程序员也能编写/修改数据处理逻辑？

**核心做法：**

```yaml
pipeline:
  - fetch: { url: "${{ args.url }}" }
  - select: "data.items"
  - filter: "item.active"
  - map: { id: "${{ item.id }}", name: "${{ item.name }}" }
  - limit: 20
```

**适用场景：**
- 数据 ETL
- API 聚合
- 报表生成

---

### 模式4：Step Registry 扩展模式

**解决的问题：** 如何让核心代码不依赖具体实现，同时支持无限扩展？

**核心做法：**

```typescript
// 1. 定义统一接口
type StepHandler = (page, params, data, args) => Promise<Result>

// 2. 核心代码只查表
const handler = registry.get(stepName);
const result = await handler(page, params, data, args);

// 3. 扩展只需注册
registry.register('my-step', myHandler);
```

**适用场景：**
- 插件系统
- 工具链（编译器、构建器）
- 协议适配器

- **Pipeline Executor**：顺序执行 YAML 步骤
- **Step Registry**：可扩展的步骤字典

## 对我的启发

### 如果我在做类似项目

**我会借鉴：**
1. **Strategy 字段设计**——把权限/安全策略编码进配置声明，而非运行时判断
2. **运行时零 AI 成本**——AI 只在生成阶段介入，运行时纯确定性执行
3. **YAML Pipeline**——声明式覆盖 80% 场景，代码扩展剩余 20%
4. **Step Registry**——用注册表模式实现无限扩展，核心代码不改动

**我会改进：**
1. **增加类型提示**——YAML 模板 `${{ }}` 目前无类型检查，可以在 schema 层面增加
2. **支持更多数据源**——目前主要是 HTTP 和浏览器，可以增加数据库、文件系统
3. **增加调试模式**——可视化每一步的数据流转，方便排查问题

### 这个项目教会我的 3 件事

1. **AI 的最佳位置是"生成配置"，而非"运行时决策"**——配置是静态的、可审查的、零运行时成本的

2. **复用用户已有的东西，比让用户给你东西更优雅**——复用 Chrome 登录态比管理密码更安全、更简单

3. **声明式 + 扩展点是黄金组合**——声明式覆盖常见场景，扩展点覆盖特殊需求，两者缺一不可
