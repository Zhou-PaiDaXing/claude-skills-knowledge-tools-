# 2.2 Provider / Transport 双层抽象 ⭐ (v0.11)

> 📚 来源:[[hermes-agent-learning-plan#2.2 Provider / Transport 双层抽象 ⭐ (v0.11)|学习计划 · 阶段二 · 2.2]]
> 🧭 相关:[[2.1 Tool Registry 自注册模式]] · [[3.3 OpenAI 兼容本地代理]]

---

## 设计思路

**核心思想**：将"提供商的 quirks（特性/怪癖）"与"API 调用路径"分离

| 层 | 职责 | 代码位置 |
|---|---|---|
| **ProviderProfile** | 声明提供商的**静态配置**和**quirks** | `providers/base.py` |
| **ProviderTransport** | 实现**数据转换**和**API 调用路径** | `agent/transports/base.py` |

---

## ProviderProfile - 声明式配置层

```python
@dataclass
class ProviderProfile:
    # ── 身份标识 ─────────────────────────────────────────────
    name: str                    # 提供商名称，如 "nvidia"
    api_mode: str = "chat_completions"  # 使用的 API 模式
    aliases: tuple = ()          # 别名列表

    # ── 认证与端点 ──────────────────────────────────────────
    env_vars: tuple = ()         # 需要的环境变量
    base_url: str = ""           # API 基础 URL
    models_url: str = ""         # 模型列表端点
    auth_type: str = "api_key"   # 认证类型

    # ── 客户端级 quirks ────────────────────────────────────
    default_headers: dict[str, str] = field(default_factory=dict)

    # ── 请求级 quirks ──────────────────────────────────────
    fixed_temperature: Any = None           # 固定温度（或 OMIT_TEMPERATURE）
    default_max_tokens: int | None = None   # 默认最大 token 数
    default_aux_model: str = ""             # 辅助任务用的廉价模型

    # ── Hooks（可被子类重写）───────────────────────────────
    def prepare_messages(self, messages): ...
    def build_extra_body(self, **context): ...
    def build_api_kwargs_extras(self, **context): ...
    def fetch_models(self, **kwargs): ...
```

**关键特性**：
- **声明式**：只描述"是什么"，不实现"怎么做"
- **可组合**：通过 `default_headers`、`fixed_temperature` 等字段声明 quirks
- **可扩展**：通过 hooks 支持复杂逻辑

## ProviderTransport - 数据转换层

```python
class ProviderTransport(ABC):
    """ProviderTransport 拥有 API 调用路径的数据转换逻辑"""

    @property
    @abstractmethod
    def api_mode(self) -> str:
        """返回此 Transport 处理的 api_mode，如 'anthropic_messages'"""
        ...

    @abstractmethod
    def convert_messages(self, messages: List[Dict], **kwargs) -> Any:
        """将 OpenAI 格式消息转换为提供商原生格式"""
        ...

    @abstractmethod
    def convert_tools(self, tools: List[Dict]) -> Any:
        """将 OpenAI 格式工具定义转换为提供商原生格式"""
        ...

    @abstractmethod
    def build_kwargs(self, model, messages, tools, **params) -> Dict:
        """构建完整的 API 调用 kwargs（主要入口）"""
        ...

    @abstractmethod
    def normalize_response(self, response: Any, **kwargs) -> NormalizedResponse:
        """将提供商原生响应标准化为 NormalizedResponse"""
        ...
```

**关键特性**：
- **面向 API 模式**：每个 `api_mode` 对应一个 Transport
- **数据转换**：负责 OpenAI 格式 ↔ 提供商原生格式的转换
- **响应标准化**：将不同提供商的响应统一为 `NormalizedResponse`

## 协作方式

```
┌─────────────────────────────────────────────────────────────────┐
│                        AIAgent (run_agent.py)                    │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │  1. 根据 provider 名称获取 ProviderProfile                      ││
│  │     profile = get_provider_profile("nvidia")                 ││
│  │                                                              ││
│  │  2. 根据 api_mode 获取 ProviderTransport                       ││
│  │     transport = get_transport(profile.api_mode)              ││
│  │                                                              ││
│  │  3. 调用 Transport.build_kwargs()，传入 Profile               ││
│  │     kwargs = transport.build_kwargs(                        ││
│  │         model=model, messages=messages, tools=tools,        ││
│  │         provider_profile=profile,  # ← Profile 传入          ││
│  │         **params)                                           ││
│  │                                                              ││
│  │  4. 执行 API 调用                                              ││
│  │     response = client.chat.completions.create(**kwargs)      ││
│  │                                                              ││
│  │  5. 标准化响应                                                ││
│  │     normalized = transport.normalize_response(response)      ││
│  └─────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
```

## 注册机制

### ProviderProfile 注册

```python
# providers/__init__.py
_REGISTRY: dict[str, ProviderProfile] = {}

def register_provider(profile: ProviderProfile) -> None:
    _REGISTRY[profile.name] = profile
    for alias in profile.aliases:
        _ALIASES[alias] = profile.name

def get_provider_profile(name: str) -> ProviderProfile | None:
    if not _discovered:
        _discover_providers()  # ← 懒加载
    return _REGISTRY.get(_ALIASES.get(name, name))
```

### ProviderTransport 注册

```python
# agent/transports/__init__.py
_REGISTRY: dict = {}

def register_transport(api_mode: str, transport_cls: type) -> None:
    _REGISTRY[api_mode] = transport_cls

def get_transport(api_mode: str):
    cls = _REGISTRY.get(api_mode)
    return cls() if cls else None
```

## 懒加载发现机制

```python
def _discover_providers() -> None:
    """Populate the registry by importing every provider plugin.

    Order:
      1. Bundled plugins at ``<repo>/plugins/model-providers/<name>/``
      2. User plugins at ``$HERMES_HOME/plugins/model-providers/<name>/``
      3. Legacy per-file modules at ``providers/<name>.py`` (back-compat)

    Each step imports its plugins, which call ``register_provider()`` at
    module-level. Later steps win on name collision.
    """
    global _discovered
    if _discovered:
        return
    _discovered = True

    # 1. Bundled plugins
    for child in _BUNDLED_PLUGINS_DIR.iterdir():
        _import_plugin_dir(child, "bundled")

    # 2. User plugins (可以覆盖 bundled)
    for child in user_dir.iterdir():
        _import_plugin_dir(child, "user")

    # 3. Legacy modules
    for modname in pkgutil.iter_modules(_pkg.__path__):
        importlib.import_module(f"providers.{modname}")
```

## 架构优势

| 优势 | 说明 |
|---|---|
| **单一职责** | Profile 只声明配置，Transport 只负责转换 |
| **可测试性** | 可以独立测试 Profile 的配置和 Transport 的转换逻辑 |
| **可扩展性** | 添加新提供商只需注册 Profile，添加新 API 模式只需实现 Transport |
| **消除布尔标志** | 之前需要 20+ 个布尔标志，现在全部来自 Profile |
| **渐进式迁移** | Transport 可以返回 `None`，允许逐步迁移到新的架构 |

## Provider 懒加载 vs Tool 即时加载

| 维度 | Provider（懒加载） | Tool（即时加载） |
|---|---|---|
| **加载时机** | 第一次调用 `get_provider_profile()` 时 | `model_tools.py` 被导入时立即加载 |
| **原因** | Agent 只用一个 Provider，无需全部加载 | Agent 启动时需要知道所有工具的 schema 发给 LLM |
| **性能影响** | 节省启动时间 | 确保 LLM 一开始就能选择任何工具 |
| **覆盖机制** | 后注册的覆盖先注册的（last-writer-wins） | 注册即固定，不可覆盖 |

## 总结

这个双层抽象的核心价值在于：

1. **ProviderProfile** = "这个提供商有什么 quirks"（声明式配置）
2. **ProviderTransport** = "如何转换数据以适应这个 API 模式"（数据转换逻辑）

两者正交：一个提供商可能有多个 API 模式（需要多个 Transport），一个 API 模式可能被多个提供商使用（共享同一个 Transport，但使用不同的 Profile）。
